1. System overview

StepTracker is a serverless single-page application (SPA) for tracking daily step counts, viewing leaderboards, and monitoring fitness statistics. It runs entirely on free-tier cloud services with no backend server to manage.

Frontend

React SPA built with Vite, deployed as static files to GitHub Pages. All business logic runs in the browser.

Backend

Firebase provides authentication, database (Firestore), and hosting rules. No custom server code is needed.

Hosting

GitHub Pages serves the static build. GitHub Actions automates the CI/CD pipeline on every push to main.

Cost

Entirely free tier: Firebase Spark plan (unlimited auth, 1GB Firestore, 50K reads/day) + GitHub Pages (free for public repos).

3. Architecture diagram

The application follows a serverless architecture where the browser communicates directly with Firebase services.

How it works

The browser loads the React SPA from GitHub Pages. Once loaded, all data operations happen directly between the browser and Firebase using the Firebase JavaScript SDK. There is no intermediary server — Firebase Auth handles authentication, and Firestore security rules enforce access control at the database level.

5. Data model

Firestore uses two collections. Documents are schemaless, so new fields can be added without migration.

users/{uid}

One document per registered user. The document ID is the Firebase Auth UID.

FieldTypeDescription
emailstringUser's email address
displayNamestringFull name from auth provider
nicknamestringUser-chosen display name
photoURLstringProfile photo URL (from Google SSO)
heightnumber|nullHeight in cm
weightnumber|nullCurrent weight in kg
stepGoalnumberDaily step target (default: 10,000)
targetWeightnumber|nullGoal weight in kg
statusstringpending | approved | rejected
rolestringuser | admin
createdAttimestampRegistration timestamp
updatedAttimestampLast profile update

activities/{userId_date}

One document per user per day. The document ID is {userId}_{YYYY-MM-DD}, ensuring uniqueness and enabling direct lookups.

FieldTypeDescription
userIdstringOwner's Firebase Auth UID
datestringActivity date in YYYY-MM-DD format
stepsnumberStep count for the day
updatedAttimestampLast update timestamp
Extending the data model: To add new activity types (calories, distance, sleep), add fields to the activities document. Firestore is schemaless, so existing documents are unaffected. Similarly, new profile fields can be added to the users collection without migration.

6. Authentication flow

Key behaviours:

  • The admin email (jithankurian@gmail.com) is auto-approved with admin role on every login
  • All other users start with status: pending and must be approved by the admin
  • The PrivateRoute component redirects unapproved users to the pending screen
  • A Firestore security rule allows the admin email to always update their own status/role (bootstrap rule)

7. Security rules

Firestore security rules run server-side and are the primary access control mechanism. They are maintained as a reference copy in firestore.rules and must be manually deployed to the Firebase Console.

Users collection

OperationRule
ReadAny authenticated user (needed for leaderboard)
CreateAuthenticated user can create their own document only
Update (own)Can update own profile, but not status or role fields
Update (admin bootstrap)Admin email can always update own status/role
Update (admin)Users with role: admin can update any user document

Activities collection

OperationRule
ReadApproved users only
Create/UpdateApproved users can write their own activities only (userId must match auth UID)

9. Manual deployment

If you need to deploy without GitHub Actions:

# 1. Create a .env file with your Firebase config
cp .env.example .env
# Edit .env with your values

# 2. Install dependencies
npm install

# 3. Build the production bundle
npm run build

# 4. The dist/ folder now contains the deployable static files.
#    Upload to any static hosting service:
#    - GitHub Pages (drag to repo or use gh-pages branch)
#    - Netlify (drag dist/ folder to deploy)
#    - Vercel, Cloudflare Pages, or any static host
SPA routing: GitHub Pages returns 404 for deep links (e.g. /leaderboard). The public/404.html file handles this by redirecting to index.html with the original path preserved as a query parameter. The index.html script restores the path via history.replaceState. If deploying to a different host, configure server-side redirects to serve index.html for all routes instead.

10. Scaling considerations

The current architecture handles a small to medium user base on free tier. Here's what to consider as the app grows:

ConcernCurrent approachAt scale
Leaderboard computation Client-side aggregation of all activities Move to Cloud Functions that precompute weekly/monthly aggregates
Firestore reads 50K free reads/day (Spark plan) Upgrade to Blaze plan (pay-as-you-go, still has free tier)
Profile photos Google SSO avatar only Add Firebase Storage (Blaze plan) or base64-in-Firestore for small images
Activity queries Fetch all activities in date range, filter client-side Add Firestore composite indexes for server-side userId+date filtering
Real-time updates Manual refresh / page navigation Use Firestore onSnapshot listeners for live leaderboard updates
Admin notifications Admin checks dashboard manually Add Cloud Functions to send email/push on new user registration
